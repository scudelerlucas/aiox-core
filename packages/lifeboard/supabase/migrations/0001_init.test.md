## Como rodar

O script `0001_init.test.sql` é auto-contido: cria um stub `auth.uid()` (o Supabase provê
`auth` em produção; localmente não há GoTrue), aplica `0001_init.sql` via `\i` e roda 10 testes.

### Contra um Postgres local efêmero (o que foi feito nesta rodada)

```bash
# 1. cluster efêmero (usuário postgres do SO; NÃO o Supabase remoto)
initdb -D /var/lib/postgresql/lifeboard_test -U postgres --auth=trust
pg_ctl -D /var/lib/postgresql/lifeboard_test -o "-p 5433 -k /tmp -c listen_addresses=''" start

# 2. banco de teste + suíte
psql -h /tmp -p 5433 -U postgres -c "create database lifeboard_test;"
cd packages/lifeboard/supabase/migrations
psql -h /tmp -p 5433 -U postgres -d lifeboard_test -v ON_ERROR_STOP=0 -f 0001_init.test.sql
```

## Por que o anti-ciclo cobre as duas representações

O grafo carrega a mesma aresta de precedência em dois campos: `predecessor_ids` (aresta
`x → esta`) e `successor_ids` (aresta `esta → y`). O trigger `lifeboard_check_task_dag()`
monta o conjunto de arestas **combinando os dois campos** de todas as linhas (usando os
valores `NEW` para a linha alterada) e faz uma travessia recursiva (CTE) a partir dos
sucessores diretos de `NEW`. Se `NEW.id` for alcançável de volta, há ciclo → `RAISE`.
Como `NEW` é a única linha alterada, qualquer ciclo novo passa por ela; `UNION` deduplica
nós, então a recursão sempre termina. Testes 3, 4 e 5 exercitam predecessor, auto-referência
e successor respectivamente.

## Notas para a revisão manual da manhã

- **Não aplicado no Supabase real.** Aplicar só após revisão; se o CLI acusar operação
  destrutiva, usar `--confirm-destructive` (kill-switch nº 1 do PRD).
- **Stub `auth.uid()` é só de teste local** — NÃO faz parte de `0001_init.sql`. Em produção
  o Supabase provê `auth.uid()`.
- **`ON DELETE CASCADE`** nas FKs (`source_id`/`project_id`): apagar uma `source` remove seus
  `projects`/`tasks`/`sync_log`. É intencional (ciclo de vida da fonte), mas é destrutivo —
  confirmar se é o comportamento desejado antes de operar deleções em prod.
- **`created_at`** foi adicionado às 4 tabelas como baseline de auditoria (princípio Dara);
  não estava no PRD §7 mas é aditivo e inofensivo. Remover se o revisor preferir aderência
  estrita ao PRD.
